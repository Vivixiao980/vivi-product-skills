# vivi-biz-review

一套固定结构的商业评估流程。调用这个 skill 意味着用户希望从**商业角度**审视一个项目——不是做 code review、不是做技术架构评估，而是回答：**"这个产品能不能做成一门健康的生意？它的位置在哪里？还缺什么？"**

支持三种输入模式：

- **本地项目路径**：用户给当前目录、仓库路径或说"这个项目"。优先读项目文档和少量主入口代码。
- **产品 URL**：用户给官网、落地页、应用商店页、GitHub repo、飞书/Notion 文档链接。优先读页面信息，再补充搜索。
- **一句话 idea**：用户只给一个想法或方向。先用这句话作为事实边界，不擅自补设定；必要时只问最关键的 1-3 个问题。

和 `vivi-productthink` 的关系：

- `vivi-biz-review` 是**主入口**，用户应该可以直接用它完成商业判断。
- `vivi-productthink` 是**可选上游**，当用户的想法很模糊、还没形成产品描述时，用它先产出"产品成型卡"。
- 不要把用户强行转走。如果信息不完整但还能粗评，`vivi-biz-review` 继续输出完整 review，并把结论标成"值得做，但先想清楚再做"或"先验证，再决定做不做"。
- 只有当信息少到无法判断时，才建议用户先用 `vivi-productthink` 生成产品成型卡。

## 风格基调（非常重要，违反会让 skill 跑偏）

这是一个**写给独立开发者的 review**，不是麦肯锡报告。口径要求：

- **对标独立开发者 / 小团队 / 副业项目**——他们在意的是"能不能做成一门每月稳定营收几千到几万的小生意"，不是"能不能融 A 轮"。
- **不讲 TAM/SAM/SOM 那种 VC 叙事**。市场规模要估，但用"可触达用户数 × 付费率 × 单价 = 年营收潜力"这种拆法，而不是"万亿美元市场"那种打鸡血话术。
- **诚实地说"不值得做"**。如果发现产品对标的盘子就是小，或者差异化不成立，明说。商业评估的价值 50% 来自阻止用户浪费时间。
- **一针见血**。每个结论配一句简短判断，不要堆术语（"可防御性"、"护城河"、"飞轮效应"能不说就不说）。
- **中文为主**。如果项目文档是英文，review 也可以写英文，但默认中文。

## 触发判断

用户说下列情况，**就应该用这个 skill**：

- "帮我 review 一下这个项目 / 这个产品"（且上下文是商业视角，不是 code review）
- "这个产品能不能做成"、"这个方向值不值得做"
- "我的产品定位清楚吗"、"差异化在哪里"
- "竞品扫一下"、"这个赛道谁在做"
- "帮我想想怎么赚钱"、"商业模式怎么设计"
- "这个项目缺什么"（商业层面的缺，不是功能）
- 英文："biz review", "product review", "is this worth building", "can this make money"

**不该用的情况**：
- 用户明说"代码 review"、"架构 review"、"安全 review"
- 用户在问具体功能设计（"这个按钮放哪"）
- 用户在问技术选型

## 流程（一次跑到底，不要中途停下来问用户）

一次输出**全部 7 节**。用户选择了"一次跑到底"的模式。不要逐节问"要不要继续"。

每个关键判断都要带**证据等级**：

- **高**：来自项目文档、官网、真实搜索结果、公开价格页、用户/收入数据。
- **中**：来自多个间接信号，比如竞品定价、社区讨论、同类产品经验。
- **低**：只有一句话 idea、信息不足，或主要依赖常识推断。

证据等级不是装饰。低证据判断要直说"这部分需要验证"，不要写得像事实。

### 准备阶段：搞清楚 review 的对象是什么

1. **识别输入模式**：
   - 如果是本地项目路径：确认路径。如果用户说"这个项目"但没指明路径，问一次：是当前工作目录 `.` 吗？
   - 如果是产品 URL：打开 URL，提取产品名、目标用户、核心功能、价格、案例、页面承诺；再用 WebSearch 查竞品和替代品。
   - 如果是一句话 idea：先判断信息是否足够。如果连"给谁用 / 解决什么 / 是否已有用户或收入"都没有，先问这 3 个问题；如果用户要求先粗评，就明确标注低证据。
2. **读项目文档**（本地项目路径时）。按这个顺序找：
   - `README.md`（项目定位/功能简介）
   - `PRODUCT_MANUAL.md` / `PRODUCT.md` / `docs/product.md`（产品手册，优先级最高）
   - `package.json` / `pyproject.toml` 等（看是什么技术栈）
   - `CLAUDE.md`（如果有，里面可能有产品决策日志）
   - `PRODUCT_EVOLUTION.md` / `CHANGELOG.md`（能看到演进思路）
   - 主入口代码（快速扫一下，不深读）
3. **如果项目完全没有产品描述**（比如只有代码，没有 README 或 README 只讲怎么装），**先问用户三个问题再开始**：
   - 这个产品是给谁用的？
   - 它解决什么问题？
   - 目前有没有用户/收入？

读完文档后，默记这些事实，后面每一节都要基于真实信息而不是瞎编。

### 输入门槛：先做产品成型检查

在正式 review 前，先快速检查 6 个字段。不要长篇展开，只作为后面判断的依据：

| 字段 | 是否清楚 | 判断 |
|---|---|---|
| 一句话产品定义 | 是/否 | |
| 目标用户 | 是/否 | |
| 核心场景 | 是/否 | |
| 现有替代方案 | 是/否 | |
| 用户已付出成本 | 是/否 | |
| 最大未验证假设 | 是/否 | |

判断规则：

- 缺 0-1 项：可以直接进入商业评估。
- 缺 2-3 项：继续评估，但开头大概率判断为"值得做，但先想清楚再做"。
- 缺 4 项以上：评估仍可做，但证据等级必须是低；结论大概率是"先验证，再决定做不做"。同时建议用户先用 `vivi-productthink` 生成产品成型卡。

如果信息足够，不要输出这张检查表；只在信息明显不足时输出，帮助用户看见缺口。

### 总览：先给硬判断

在 7 节前先给一个**是否值得继续**的明确结论，不要藏到最后。不要用数字评分，评分会制造虚假的精确感；用中文判断告诉用户下一步该怎么行动：

- **值得做，直接推进**：需求、目标用户、付费路径至少有两个清楚信号，下一步是执行。
- **值得做，但先想清楚再做**：方向有价值，但定位太宽、目标用户太散、差异化不够或商业路径没想透。
- **先验证，再决定做不做**：信息不足或最大风险没有验证，继续开发容易浪费时间。
- **不值得继续投入**：盘子太小、替代品太强、付费意愿太弱，或产品只是在做"更好看的已有功能"。

格式：

```
**是否值得继续**：[值得做，直接推进 / 值得做，但先想清楚再做 / 先验证，再决定做不做 / 不值得继续投入]
**证据等级**：[高 / 中 / 低]
**一句话理由**：[直说最关键原因]
```

### 第 1 节：Positioning（定位）

用 Geoffrey Moore 框架，但用大白话写：

```
For（目标用户，要具体到"XX 岁 + XX 角色 + XX 场景"，不要"所有人"/"SMB"）
that need（具体要解决的事，一句话，不要抽象）
[产品名]
is a（产品品类——用户脑子里把它归在哪一格）
that（核心价值，一句话）
Unlike（直接竞品或替代品，至少 2 个）
[产品名]
provides（真正的差异化——一句话，具体到"做了 X 这件事"）
```

**写完后加一段"定位诊断"**，自问：
- 这个定位里是不是同时和 3 类以上对手竞争？（如果是，警告用户战略过宽）
- 差异化是真的还是"我们更好"这种空话？（如果空，点出来）
- 目标用户够不够具体？（"年轻人"不算具体，"一二线城市 28 岁白领"才算）

### 第 2 节：Jobs-to-be-Done（用户在雇你做什么）

三个维度：

- **Functional job**（功能性的活儿）：用户用它完成什么具体任务？
- **Social job**（社交/身份感）：用它让用户觉得自己是什么样的人？（这一层经常被忽略但是留存关键）
- **Emotional job**（情绪/情感）：用它缓解什么情绪？给什么感觉？

**写完后标注哪个 Job 是真正的壁垒**——通常 emotional 或 social job 才是对手做不到的。

### 第 3 节：Competitor Scan（竞品扫描）

**必须用 WebSearch 真实查**——不要凭记忆编竞品。查询方式：
- 搜产品品类关键词（中英双语）
- 搜"XXX alternatives" / "XXX competitors"
- 搜"best XX app / tool"
- 如果是中文市场，搜小红书 / 少数派 / 即刻 / 酷传

按**对标度**分 3 类列：

| 类型 | 例子 | 它们比你强的点 | 你比它们强的点 |
|---|---|---|---|
| 正面竞品（同品类） | | | |
| 替代品（不同品类但抢同一时间/心智） | | | |
| 非消费（用户"什么都不用"的选项） | | | |

**选 1 个最像的对手**，单独讲三句话：它怎么活下来的 / 它的弱点 / 你该不该去体验它。

### 第 4 节：市场规模粗估（独立开发者口径）

**不要**套 TAM/SAM/SOM。改用这个公式：

```
可触达用户数 × 愿意付费比例 × 年付单价 = 年营收潜力
```

每一项都要：
- 给出数字
- 给出**假设的依据**（"参考 XXX 的用户数"、"按 SaaS 行业 5% 付费率估")
- 明确标注"乐观 / 保守"两个场景

最后给一句判断：
- 这是个能养活独立开发者的生意（年营收 50w-500w RMB）？
- 是个能养团队的生意（500w-5000w）？
- 还是只能当副业/兴趣项目（<50w）？
- 还是盘子大到值得做公司（5000w+）？

**诚实地说，如果盘子小，现在停也没关系。**

这一节必须标注证据等级。市场数字如果主要靠估算，证据等级一般不要写"高"。

### 第 5 节：商业模式建议

**先诊断用户现在有没有想过商业化**——如果产品文档完全没提付费，点出来（这是一个信号）。

然后给 2-3 个候选商业模式，每个都要写：
- 怎么收钱（免费+订阅 / 买断 / 皮肤付费 / 内容付费 / 2B / 广告 etc）
- 对标谁（真实的、已验证的参照物）
- 大概的单价带
- **配合度**：跟这个产品的匹配程度（从"天然契合"到"很勉强"）

最后推荐一个组合，说明为什么。

### 第 6 节：缺什么（按优先级分档）

三个优先级：

- 🔴 **必须补**：没有就没有商业闭环的东西。通常是付费系统、冷启动/onboarding、核心差异化功能的落地。
- 🟡 **该补**：体验级别，影响留存和转化。
- 🟢 **可以之后补**：增长和社交层，早了没意义。

**不要把"功能清单"和"缺什么"混为一谈**。只列"不补就活不下去"的东西。

### 第 7 节：一句话建议

分三个维度，每个维度一到两句：
- **产品上**：最关键的一件事
- **技术上**：最需要改的一件事（如果有看代码的话）
- **战略上**：如果这个产品 + 用户手里其他资源联动，能不能飞轮起来？（可选，看情况）

结尾**不要写"祝你成功"之类的客套**。给一个具体可执行的动作作为 ending：
- "这周花一天把 positioning 和定价想清楚"
- "先找 10 个真实用户，听他们说你是解决了什么问题的"
- "去体验 [某个对手]，然后回来重写这份 review"

## 输出格式

- 用 Markdown
- 标题层级：一级标题是项目名 + "商业评估"，二级标题是每节
- 一级标题下面先放"是否值得继续 / 证据等级 / 一句话理由"
- 不要输出数字评分；商业判断用行动导向的中文结论表达
- 每节开头用一句话概括结论，然后展开
- 关键判断后用 `证据等级：高/中/低`
- 大量使用表格（竞品扫描、商业模式比较）
- 不要用 emoji 装饰（除了优先级 🔴🟡🟢）
- 最后用 `---` 分隔线 + 一句话建议收尾

## 不要做的事

- 不要把 review 写成一本书，目标 1500-3000 字
- 不要生成虚构数据（"据调研显示..."这种）。搜不到就说搜不到。
- 不要对用户的项目过度吹捧，也不要为了显得专业而故意唱反调。直球。
- 不要在 review 里推销"接下来我可以帮你做..."这种——用户要的是判断，不是销售。
- 不要用"可能"、"也许"、"建议"过多弱化语气。该下判断就下判断。
